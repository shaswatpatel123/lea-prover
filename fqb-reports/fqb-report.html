A minimal agent on FormalQualBench

April 2025 · Lea v1 + Gemini 3.1 Pro · github.com/chinmayhegde/lea-prover

We ran Lea, a minimal theorem-proving agent (~300 lines of Python, 6 tools, one system prompt), on FormalQualBench — 23 graduate-level theorems in Lean 4 spanning topology, algebra, analysis, combinatorics, and number theory. These are results at the level of a math PhD qualifying exam whose proofs are not in Mathlib and must be constructed from scratch.

The agent has no proof decomposition, no tree search, no reinforcement learning. It runs a single loop: write a proof, compile, read errors, edit, repeat. The model decides everything.

Failure modes

The 21 failures cluster into four patterns.

Search spiral (7/21 failures)

Problems: BanachStoneTheorem, BurnsidePrimeDegreeTheorem, GleasonKahaneZelazkoTheorem, Hilbert17thProblem, JordanCycleTheorem, PontryaginDuality, SchauderFixedPointTheorem.

The agent spent the majority of its turns grepping Mathlib source files — 244 grep/find calls across these problems vs. 39 compilation attempts. It searched for the theorem pre-built in Mathlib, didn't find it, and concluded the theorem was "unprovable in current Mathlib." This misunderstands the benchmark: FQB specifically tests theorems whose statements use Mathlib types but whose proofs must be constructed from scratch.

Banach-Stone is the clearest example: 99 turns, 67 greps, 10 compilations, zero progress. The agent tried queries like grep -rn "BanachStone", grep -rn "≃ₗᵢ", grep -rn "homeomorphOfAlgEquiv", and even curl to the GitHub search API. It never committed to a proof structure.

Near-miss (7/21 failures)

Problems: BorsukUlamTheorem, DLOQuantifierElimination, MaynardTaoBoundedPrimeGaps, ParisHarringtonPrinciple, SkolemMahlerLechTheorem, TernaryGoldbachTheorem, VonNeumannDoubleCommutantTheorem.

The agent built a proof structure — definitions, intermediate lemmas, the right imports — but left one or two sorrys unfilled. It could not close the last subgoal and had no mechanism to reconsider whether the decomposition itself was wrong.

DLO Quantifier Elimination is the extreme case: 209 turns and $19 in API costs, writing 77 files and running 69 lean_check calls. The agent was actively working — not searching — but kept hitting the same stuck subgoals without trying a different decomposition.

Gave up early (5/21 failures)

Problems: CollatzMapAlmostBoundedValues, ColorfulCaratheodoryTheorem, JordanDerangementTheorem, KakeyaTheorem3D, RungeTheorem.

The agent recognized the theorem was deep and stopped within 5 turns. For some (Kakeya 3D, Collatz) this is reasonable — these are Fields Medal-level results. For others (Runge's theorem) it's premature. The model's calibration of "what's feasible" is noisy.

Wrong path (2/21 failures)

Problems: ErdosDiscrepancyProblem, GreenTaoTheorem.

The agent wrote a proof file to a different path than expected, or never created the output file. An infrastructure issue, not a mathematical one.

What it solved

DeBruijnErdos

28 turns · 196s · $0.34

Combinatorics / graph theory

The De Bruijn-Erdős theorem: if every finite subgraph of an infinite graph is k-colorable, then the whole graph is k-colorable. A compactness argument. The agent found the right Mathlib infrastructure and closed the proof in 28 turns.

QuillenSuslinTheorem

60 turns · 537s · $0.96

Algebra / commutative algebra

The Quillen-Suslin theorem (Serre's conjecture): every finitely generated projective module over a polynomial ring is free. A major result in commutative algebra. The agent worked through 60 turns, building up the necessary algebraic machinery.

JordanDerangementTheorem (standalone run)

38 turns · 245s · $0.48

Algebra / group theory

Jordan's derangement theorem: a finite transitive permutation group on a nontrivial set contains a fixed-point-free element. Solved in a standalone run using a Burnside's lemma counting argument (68 lines of Lean). Did not reproduce in the batch eval due to nondeterminism.

Observations

The model knows the mathematics. In almost every case, the agent's final message correctly identifies the theorem, its history, and the proof strategy in natural language. The failure is in translation to Lean, not in mathematical understanding.

The dominant failure is strategic, not technical. 14 of 21 failures are search spirals or near-misses — problems where the agent either never committed to a proof structure or committed to the wrong one and couldn't adapt. A decompose-then-prove loop with reflection would directly address both.

Cost is heavy-tailed. One problem (DLO Quantifier Elimination, 209 turns) consumed 33% of the total budget. A turn budget or spiral detection would dramatically reduce cost with minimal impact on solve rate.

Nondeterminism matters. Jordan's derangement theorem was solved in one run and failed in another. The agent's success on any given problem is stochastic. Best-of-N sampling at the outer level would improve the effective solve rate.

"Not in Mathlib" ≠ "unprovable." The agent frequently concluded that a theorem was impossible because it couldn't find it in Mathlib. FQB specifically targets theorems that require building proofs from scratch. The prompt must make this expectation explicit.
