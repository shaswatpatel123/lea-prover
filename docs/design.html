Lea — Architecture

A config-driven Lean-proving agent with one config-in / event-out contract. The agent is the product; the CLI, a UI, and eval are consumers of the same core.

See decisions.md for the rationale and the mini-swe-agent mapping.

Architecture flow

Consumers

cli.py
--config + flag overrides
Teammate UI
renders events live
eval/ harnesses
batch, collect transcript
→

Config in

default.yaml
single source of defaults
config.load_config
base + --config overlay
validation.validate_config
I/O-free, typed errors → LeaConfig
→

Agent core

agent.run_events
generator loop — yields events; owns MCP lifecycle
providers.stream
LiteLLM, stream or blocking
registry.build_toolset
active tools: built-ins + custom + MCP
prompt.load_system_prompt
base variant + skills
→

Events out

typed event stream
TextDelta · ToolCall · Usage · Finished
render_to_stdout
CLI transcript + per-turn cost
UI / eval consumers
same events, own rendering

Key property: the LiteLLM engine swap and the streaming/blocking choice never touch the loop or the consumers — providers.stream() always yields the same event types, so everything to the right of it is invariant. The UI can also call validate_config() directly on a payload (dashed path — no disk, no agent run).

Extension points — everything is config

Three sources feed one tool registry; a fourth feeds the prompt. The loop is unchanged — it asks build_toolset() for the active tools and gets a system prompt with skills appended.

Tool sources

tools.py built-ins
read/write/edit/lean_check/bash/search — register at import
agent.tool_modules
user modules: @tool / register() custom tools
mcp.servers → MCPManager
stdio / HTTP servers; tools register by bare name (prefix on clash)
→

Registry → loop

registry.REGISTRY
name → Tool(schema, handler)
build_toolset(config.tools)
allowlist + order; null → all registered
→

Prompt

agent.skills
markdown fragments of procedural knowledge, in order
load_system_prompt
base variant + lea.md + skills

Why this shape: MCP tools and custom tools are not special cases — they land in the same REGISTRY as the built-ins, so build_toolset, the loop, events, and transcripts are untouched. See lean-lsp-mcp.md for a worked MCP + skill setup.

Turn lifecycle

run_events → consumer
TurnStarted(turn)
run_events → providers.stream → LiteLLM
stream(model, msgs, tools, model_kwargs, streaming)
providers → run_events
TextDelta*ToolCall (+_ToolMeta)Done(usage, cost)
run_events → consumer
AssistantTextDelta*ToolCalled*UsageUpdated(in, out, cost) ← per-turn cost
if tool calls present
run each resolved handler build_toolset()[name](args) → ToolResulted, persist session, next turn
else (no tool calls, or max_turns hit)
Finished(reason, text, turns, usage, cost, transcript)  reason ∈ {completed, max_turns}

Config schema

model:
  name: gemini/gemini-3.1-pro-preview   # LiteLLM provider/model
  stream: true                          # true → live tokens; false → one blocking call
  model_kwargs:                         # open passthrough to litellm.completion
    max_tokens: 16384
agent:
  prompt_variant: default
  max_turns: null                       # null → run until done
  tools: null                           # null → all registered; or [names] to allow + order them
  tool_modules: []                      # python modules that register custom tools
  skills: []                            # markdown files injected into the system prompt, in order
mcp:
  servers: {}                           # name: {command, args, env}  (stdio)
                                        #   or {url, headers, transport}  (remote)
